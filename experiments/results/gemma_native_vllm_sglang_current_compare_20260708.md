# Gemma Native vs vLLM/SGLang Current Comparison

Date: 2026-07-08, updated 2026-07-10

## Current Verdict

WKVM is native on the measured Gemma path, but it is not yet comparable to vLLM/SGLang on throughput. The current benchmark rows report `uses_hf_transformer_forward=false`, `uses_hf_model_construction=false`, and `native_gemma_checkpoint_loader=true`, so the measured path does not call HF `Gemma4ForCausalLM` construction or HF decoder/attention `forward`. It still uses HF-adjacent metadata/helpers where useful, such as config/tokenizer handling and the Gemma rotary helper, so "no HF" here means no HF Transformer model execution/construction in the hot inference path, not zero `transformers` package contact.

The strongest same-window uniform B8 comparison is now WKVM dense no-graph at 228.219 tok/s green and WKVM token-pool auto-persistent rows plus sliding page reclaim at 213.488 tok/s, versus vLLM at 240.302 tok/s but not green under this report's memory envelope. The new token-pool B8 row is also not green under the current desktop envelope because another process holds about 4.6-4.7 GiB of VRAM, but it succeeds with a tighter 24576-slot token pool after page reclaim. The saved staggered-prompt ladder is harsher: WKVM dense reaches 259.567 tok/s green at B8, while vLLM reaches 544.177 tok/s but not green. Treat this as a partial native-engine milestone, not parity.

Latest native timing instrumentation on ctx13824/out128/B4 confirms where the token-pool path is spending time. The no-timing baseline is 137.197 tok/s; the timing-enabled run is 136.443 tok/s, so the instrumentation overhead is small enough for diagnosis. Decode-only model-forward time is 1.528221s. Inside it, token-pool attention is 0.436437s, self-attention total is 0.941640s, Q norm/RoPE is 0.158601s, K/V norm/RoPE is 0.097470s, QKV projection is 0.081026s, MLP is 0.155958s, PLE is 0.157530s, and current-token token-pool K/V writes are only 0.060766s. The pool-wide `kv_set_wall_s` is 0.375811s because it includes prefill/backfill and host-timed copy phases; it is not the main decode-only bottleneck.

Two source-matched probes were measured and rejected. The SGLang-style fused QKV RMSNorm probe reached 136.815 tok/s without timing versus the 137.197 tok/s baseline, so the patch was removed. The paged token-pool B4 probe reached 135.572 tok/s with 4445/4445 paged successes, also slower than baseline. These results point away from isolated norm fusion and simple paged dispatch toggles as the next win.

  --model /run/media/xiaol/B214449214445C0B/models/gemma/gemma-4-E4B-it \

A fresh source-boundary check of the local vLLM and SGLang trees confirms the same structural gap. vLLM's Gemma4 attention computes Q/K/V and calls generic `Attention`; the generic layer performs the unified KV-cache update, consumes runner-built block tables/slot mapping/common attention metadata, and dispatches through the selected backend and CUDA graph context. That is the next source-boundary gap, independent of throughput.

WKVM's native bridge implements explicit PyTorch Gemma layer math and a WKVM-owned routed-span cache. The production path still uses PyTorch SDPA or the single-row manual GQA path over materialized KV tensors. Distinct-cache batching is handled by exact or padded materialized merges, persistent padded decode groups, and optional CUDA graph replay. The dense `triton_dense_gqa` probe added a decode-only grouped-query Triton kernel over padded K/V tensors, but it launches only `batch * kv_heads` programs and each program scans the whole context, so it does not have vLLM/SGLang-style split-sequence parallelism and reduction over paged KV; the B8 measurement regressed to 194.030 tok/s, so it is not the current frontier. The token-pool attention path writes decode K/V into a `TokenKVPool` and consumes `kv_indptr`/`kv_indices`. It now has layer-id metadata for nonchronological full/routed readout order, so both `sliding_attention` and `full_attention` can be covered in the native no-HF path. `PagedDecodeBatchMetadata`/`build_paged_decode_metadata_from_token_slot_rows` can emit selected-window-relative `block_tables`, `block_table_lens`, selected starts, sequence lengths, and `slot_mapping` for page-aligned rows, with an explicit fixed `block_table_width` for graph-stable static buffers. The live request-table path can also build paged sliding metadata directly from `ReqToTokenTable` slices, avoiding an extra selected-row CPU list materialization before block-table construction. The latest page-table path mirrors engine-owned logical-to-physical page maps into `_token_pool_page_table_tensor`, and `ReqToTokenTable.build_paged_decode_metadata_from_page_table_tensor()` gathers fixed-width block tables from that dense tensor before falling back to dict or row-scan builders. `TokenSlotAllocator.alloc_page_block_with_ids()` and `TokenKVPool.alloc_page_block_with_ids()` let the engine reserve whole page blocks for sliding token-pool rows, and `GemmaNativeEngine` now builds live paged sliding metadata into `TokenPoolDecodeContext.paged_metadata_by_layer_type` and `paged_metadata_by_layer_id`. `wkvm/runner/gemma_token_pool_triton.py::token_pool_paged_gqa_decode` provides the grouped Triton decode primitive for that block-table metadata, and `WKVM_ENABLE_TOKEN_POOL_PAGED_TRITON=1` now routes the native serving benchmark through it. This is still not the default production path because it is an explicit opt-in gate and it covers the sliding page-aligned path while full/routed rows still use materialized/backfilled flat metadata. CUDA token-pool decode otherwise auto-dispatches to the flat grouped Triton GQA kernel in `wkvm/runner/gemma_token_pool_triton.py`, unless `WKVM_ENABLE_TOKEN_POOL_TRITON=0` or `WKVM_DISABLE_TOKEN_POOL_TRITON=1` disables it. The same Triton module now also exposes `token_pool_store_kv()` for flat slot-mapped K/V writes; host-known contiguous spans still use direct slice copies, while CUDA slot-mapped writes can use one store kernel with fallback counters instead of two PyTorch `index_copy_` calls. An opt-in `WKVM_ENABLE_TOKEN_POOL_SPLIT_TRITON=1` route adds flat split-KV decode: stage 1 launches over `(batch, kv_heads, split)` and stores per-split max/expsum/accumulators, then stage 2 reduces with log-sum-exp. A separate `WKVM_ENABLE_TOKEN_POOL_PAGED_SPLIT_TRITON=1` route adds the same two-stage structure to paged `block_tables` metadata. `WKVM_TOKEN_POOL_TRITON_MIN_SPLITS` now defaults split-KV to rows with at least four chunks, so short 1024-token sliding rows use the simpler grouped kernel while long full/routed rows still split. These source structures now match the broad split dimension used by incumbent decode paths, but the closest comparison is usually scalar/global split count plus per-row `seq_lens` bounds, not necessarily an explicit per-row split-count tensor. Unsupported shapes still fall back to the reference gather-plus-row-loop path with benchmark-visible counters. The flat Triton paths now run on the full/routed RTX 4090 shape with the wide-head `BLOCK_N=32` default, but WKVM still lacks vLLM/SGLang's integrated paged KV cache and graph decode path as the default serving path. The latest source audit also checked Triton decode dot precision. Local vLLM and SGLang grouped decode kernels do not force `input_precision="ieee"` for fp16/bf16 decode: their grouped paths either omit `input_precision` on `tl.dot` or use elementwise reductions for non-grouped paths, and SGLang casts grouped query/probability tiles toward K/V dtype. WKVM previously cast Q/K/V tiles to fp32 and hard-coded IEEE on every grouped token-pool `tl.dot`. WKVM now has `WKVM_TOKEN_POOL_TRITON_INPUT_PRECISION` and `WKVM_TOKEN_POOL_TRITON_DOT_DTYPE`; float32 keeps the exact fp32/IEEE test path, while bf16/fp16 serving can use native dot operands. Native-dot mode pads the internal grouped-head tile to 16 so Triton bf16 PV dot satisfies tensor-core shape constraints, and the split-KV workspace allocator now asks for the same block-group size as the kernel. The paired B8 persistent full-row run improves from 171.088 tok/s with fp32-dot+TF32 to 183.626 tok/s with native-dot, and the non-persistent ceiling ladder reaches 48.245/87.772/144.269/209.167 tok/s at B1/B2/B4/B8. Current token-pool/full-routed root cause after the latest work:

1. WKVM still materializes routed/full readout K/V for full/routed token-pool metadata. This is not true for every token-pool-covered decode path: covered sliding/routed placeholders can skip dense padded merge work. The remaining full/routed metadata path still depends on `NativeRoutedSpanLayer.update`, `commit_decode_token`, and `_materialize` in `wkvm/runner/gemma_runner.py` keeping `self.keys/self.values` as a dense readout; `_materialize` concatenates sink, slot means, retained spans, pending tokens, and ring tokens into dense K/V.
2. `NativeGemmaRoutedCache.merge_padded_decode` can return token-pool-covered placeholders for `NativeRoutedSpanLayer` and `NativeSlidingWindowLayer`, and `GemmaRoutedSpanRunner.decode_batch` bypasses exact dense merge when native token-pool coverage is active. The measured full/routed reference runs confirm this path: B8 has 1016 covered rows for both `full_attention` and `sliding_attention`, and B16 has 2032 rows for both types with no fallback decode calls.
3. The engine token-pool specs now describe `sliding_attention` and `full_attention` owners plus KV-shared aliases, and `TokenKVPool` allocation is lazy so full/routed specs do not allocate buffers until a layer writes into them. The default full/routed metadata path still allocates materialized-readout token slots per request, backfills those slots from `NativeRoutedSpanLayer.write_materialized_readout_to_token_pool`, appends the decode output slot, and uses `build_decode_metadata_from_token_slot_rows`. The persistent full-row path is now the automatic default when token-pool attention and persistent padded decode are active, with an explicit disable flag for A/B baselines. It keeps row-owned materialized slots and row-owned decode append slots across padded decode reuses, but it still starts from the same routed-cache materialized readout. The builder can reuse stable workspace tensors and fixed graph-padding capacity, so CUDA graph replay no longer needs to copy these metadata tensors when the shape is static. That is correct for the sink/slots/spans/pending/ring order, but it is not yet an optimized paged attention layout. The new paged metadata builder and page-aligned allocator are wired into live sliding decode, including selected-window starts, final-token `out_cache_loc`, graph-stable block-table width, and fully expired page-block reclaim, but they do not remove the full/routed materialized-readout/backfill step yet.
4. vLLM avoids the equivalent dense per-request materialization with block tables and slot mappings. In the installed vLLM source, `vllm/v1/worker/gpu/block_table.py::BlockTables` stores per-request block ids, `vllm/v1/worker/gpu/model_runner.py::prepare_attn` emits block tables plus slot mappings, and `vllm/model_executor/layers/attention/attention.py::unified_kv_cache_update` scatters new K/V by `slot_mapping`. Decode kernels consume paged K/V, e.g. `vllm/v1/attention/backends/flash_attn.py` passes `block_table` to varlen decode and writes with `reshape_and_cache_flash`.
5. SGLang uses the same structural idea with request-to-token and token-to-KV pools. `sglang/srt/mem_cache/memory_pool.py::ReqToTokenPool` maps request rows to token locations, `MHATokenToKVPool.set_kv_buffer` writes K/V by physical location, `sglang/srt/mem_cache/common.py::alloc_for_decode` allocates `out_cache_loc`, and `sglang/srt/layers/attention/flashinfer_backend.py::DecodeIndicesUpdater.call_begin_forward` builds `kv_indptr/kv_indices` for decode over the pool.

The smallest credible WKVM implementation target is not a full vLLM block-manager rewrite. The substrate now has a non-sliding token-pool placeholder layer, exact-merge bypass under token-pool coverage, type-generic mask clearing, full/shared KV layer specs, lazy token-KV buffer allocation, explicit per-row token-slot metadata with layer-id lookup for nonchronological routed layouts, selected-window-relative paged/block metadata for page-aligned rows, graph-stable paged block-table width for sliding decode, live page-aligned sliding allocation, a gated grouped Triton decode primitive for that block metadata, a primitive to mirror `NativeRoutedSpanLayer` materialized readout into token-pool slots, row-owned persistent full/routed append slots, and commit mirroring from token-pool decode output back into the authoritative routed state. The remaining target is to turn this correctness substrate into a serving-quality backend: make token-pool full/routed K/V authoritative instead of mirrored from dense routed-cache readout, make paged metadata admission/static buffers the default rather than an opt-in benchmark path, scale the paged kernel beyond the current narrow grouped decode shape, and then release dense full/routed readout storage once the token-pool representation is authoritative. The caveat remains that routed layers cannot reuse the chronological `ReqToTokenTable` blindly, because their readout order is sink/slots/spans/pending/ring rather than chronological tokens.

This is still not comparable to vLLM/SGLang as a native high-throughput inference engine. The main remaining gaps are:

1. Fused paged/ragged token-pool KV attention over WKVM state as the default fast path, without full/routed materialized-readout slot backfill or reference gather/row-loop attention.
2. Fused layer kernels where the model path supports them: Q/K/V norm, RoPE/KV-write, attention metadata, and MLP.
3. Loader parity with incumbent stacks: packed module construction, sharded checkpoints, quantized weights, tensor parallel, and MoE.
4. Scheduler/runtime parity: prefix caching, paged KV block admission/retraction, preemption, speculative lookahead, async/disaggregated KV hooks, and generalized CUDA graph dispatch.
5. A scheduler-owned or cache-manager-owned token/page KV lifecycle. WKVM has moved more decode metadata, current decode-batch state, block-table, workspace, row-lifecycle, row-planning, full-attention batch preparation, materialized writer coordination, current-token KV-write ordering, attention dispatch selection, and Triton launch accounting behind backend-shaped objects, but `GemmaNativeEngine` and `gemma_native_forward.py` still own eligibility policy, graph-buffer policy, reference fallback, timing, and hook wiring while vLLM/SGLang allocate request/KV blocks before execution and hand stable metadata buffers to attention backends.
6. Consistent dummy/padding semantics for request rows and token/page slots. SGLang reserves row/slot/page 0 as a dummy sink; WKVM currently reserves state slot 0 while token-pool padding uses a different convention.
7. Broader quality gate for packed projection; the current packed path is memory-fixed but still fails strict logits parity.
8. Production sampler/logits processors beyond the current greedy benchmark path.

The Gemma engine now uses a batched greedy sampler helper at the engine boundary. For real tensor logits it computes `argmax(dim=-1)` once per model batch and performs one CPU transfer, while keeping the row-wise fake-logits fallback for tests. This matches WKVM's generic engine shape and the local vLLM/SGLang sampler source: vLLM returns batched GPU `sampled_token_ids` from `vllm/v1/worker/gpu/sample/sampler.py` and copies outputs as tensors in `vllm/v1/worker/gpu/async_utils.py`, while SGLang's sampler uses batched `torch.argmax(logits, -1)` in `sglang/srt/layers/sampler.py`.
