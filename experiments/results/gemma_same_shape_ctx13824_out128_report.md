## 2026-07-08 native-engine retest

The best current wkvm-native same-shape row on the default HF-forward path is B8 with persistent padded decode reservation increased from 8 to 64 and CUDA graph replay enabled. It reaches 186.039 aggregate decode tok/s, 58.906 e2e output tok/s, and remains green at 16.707 GiB reserved. The same steps=64 configuration without CUDA graph reaches 183.158 tok/s at 16.686 GiB, so the graph replay win is small on the current HF-forward path.

The naive graph run with steps=8 is not a win: it captures 16 graphs, increases peak reserved memory to 20.549 GiB, and drops to 120.378 tok/s. For the current native engine, larger persistent padded reservations are more important than CUDA graph replay.

Even with the improved B8 row, wkvm-native is still not comparable to the vLLM/SGLang native transformer stacks on raw decode throughput. vLLM's tracked B8 row is 544.177 tok/s on the same ctx/out/prompt shape, but over the green memory headroom in this report. wkvm's advantage here remains bounded-memory routed-span serving, not exact full-KV transformer speed.

## 2026-07-08 no-HF-transformer-forward bridge retest

There is now an opt-in throughput path via `--use-native-gemma-forward`. This wraps the loaded HF weights with `NativeGemma4ForCausalLM`, so runner model calls do not enter `transformers.Gemma4ForCausalLM.forward`; benchmark JSON records `model_forward_backend="wkvm_native_gemma_forward_bridge"` and `uses_hf_transformer_forward=false`.

Same-shape bridge results:

- B1: 39.308 aggregate decode tok/s, 24.068 e2e output tok/s, 16.029 GiB reserved, green.
- B2: 68.373 aggregate decode tok/s, 33.871 e2e output tok/s, 16.072 GiB reserved, green.
- B4: 114.077 aggregate decode tok/s, 43.715 e2e output tok/s, 16.275 GiB reserved, green.
- B8: 165.154 aggregate decode tok/s, 50.189 e2e output tok/s, 16.566 GiB reserved, green.
- B8 with persistent padded steps=64 and CUDA graph: 179.753 aggregate decode tok/s, 51.402 e2e output tok/s, 16.881 GiB reserved, green, with 2 graph captures and 127 replays.
- SDPA attention, persistent padded steps=64, and CUDA graph ladder: B1 38.574, B2 70.670, B4 124.204, and B8 185.109 aggregate decode tok/s. The B8 row reaches 60.142 e2e output tok/s at 16.750 GiB reserved, green, with 2 graph captures and 127 replays.

This answers the HF-forward question narrowly: the bridge benchmark does not use HF transformer forward in the hot path. It still is not source-equivalent to vLLM/SGLang because it loads a Transformers Gemma object for weights/modules, keeps HF embeddings/rotary/LM-head objects, and uses eager PyTorch layer math rather than packed QKV, fused norms/MoE, and optimized attention/cache kernels. On the same B8 long-context shape, the best bridge row is 185.109 tok/s versus vLLM's tracked 544.177 tok/s. At B2, the SDPA bridge is 70.670 tok/s versus SGLang's tracked 117.488 tok/s and vLLM's 137.098 tok/s.

The SDPA bridge path is selected with `--native-gemma-attention-backend sdpa`; `manual` remains the default parity backend.

## 2026-07-08 projection backend experiment

The native bridge now exposes `--native-gemma-projection-backend {separate,qkv_packed}` and records `native_gemma_projection_backend` in engine stats and benchmark JSON. The production default remains `separate`.

The experimental `qkv_packed` path lazily concatenates Q/K/V weights for KV-owning layers and leaves shared-KV tail layers on Q-only projection. It is not accepted as a production default:

- Full-stack ctx64 parity with the default `separate` backend still passes exactly: prefill/decode/cache/logit max abs diff all `0.0`; result JSON `/run/media/xiaol/B214449214445C0B/wkvm_bench/results/native_gemma_full42_logits_parity_separate_ctx64_after_qkv_experiment.json`.
- The same full-stack check with `qkv_packed` fails the current parity bar: prefill max diff `1.25`, decode `0.625`, cache `0.2265625`, logit prefill `0.734375`, logit decode `0.84375`; result JSON `/run/media/xiaol/B214449214445C0B/wkvm_bench/results/native_gemma_full42_logits_parity_qkv_packed_ctx64.json`.
- Short ctx512/out8 SDPA A/B was not faster: `separate` reached B1 `38.247` and B2 `54.037` aggregate decode tok/s, while `qkv_packed` reached B1 `38.025` and B2 `52.210`; result JSONs `/run/media/xiaol/B214449214445C0B/wkvm_bench/results/native_gemma_forward_bridge_sdpa_separate_ctx512_out8_b1_b2_after_projection_backend.json` and `/run/media/xiaol/B214449214445C0B/wkvm_bench/results/native_gemma_forward_bridge_sdpa_qkv_packed_ctx512_out8_b1_b2_experimental.json`.

Conclusion: naive packed QKV over the live HF BF16 weights is the wrong route. A vLLM/SGLang-comparable packed projection path needs a native packed weight loader/module layout, not runtime concatenation of HF linears.
